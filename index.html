Shoot-on-the-Move targeting

FRC 5829 · design & implementation

A turretless robot normally has to stop, square up, and shoot. I built a system that lets ours score while driving. Robot velocity is decomposed into a translation vector that offsets the aim point, and an interpolating lookup table — built from time-of-flight data I measured across distances — feeds an iterative solver that converges on a compensated target. The flight time depends on distance, but the compensated distance depends on flight time, so the two are solved together by fixed-point iteration. 

public Pose2d getDynamicHubLocation() {
  Translation2d hub   = hubPose.getTranslation();
  Translation2d vel   = getFieldVelocity();      // robot velocity
  Translation2d aim   = hub;

  for (int i = 0; i < 15; i++) {     // converges in ~2–3
    double dist = aim.minus(robotPos).getNorm();
    double tof  = TOF.get(dist);          // measured lookup table
    aim = hub.minus(vel.times(tof));      // lead the goal
  }
  return new Pose2d(aim, new Rotation2d());
}

It is not perfect: sudden direction changes still cause misses, because the model assumes constant velocity over the flight time. That is a modeling limit, not a solver one, so I am moving toward Newton’s method and accounting for command latency next.

About

Two languages, two instruments, one way of thinking

I moved from China to Houston in fourth grade and grew up bilingual in English and Mandarin. I think the most interesting engineering sits at the seams between fields, which is probably why I am drawn to robotics: it is mechanical, electrical, and software all solving one problem at once.

Outside of code I play violin at the All-State level with the Houston Youth Symphony, and I have come to see a control loop and a bowed note as the same thing — both are about hitting a precise target and correcting in real time. I want to study mechanical and robotics engineering and keep building systems that move well in the real world.
